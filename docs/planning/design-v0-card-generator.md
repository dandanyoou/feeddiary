## Problem Statement

User saves content (YouTube videos, Instagram Reels, articles) constantly but never revisits them — the save buttons across platforms have become content graveyards. Beyond the personal pain, there is a wider cultural moment: people are increasingly aware that algorithms are profiling them, but no consumer-facing tool exposes this profile to the user in a digestible, shareable form. The builder's goal is to make a "flashy/cool idea program that makes myself known" — the product is partly a vehicle for the builder's identity, optimized for legend-building rather than (initially) revenue.

## What Makes This Cool

Two product surfaces, one engine:

1. **Warm taste diary (default).** Weekly card summarizing the user's saved content — top categories, taste shifts, "god-life index (갓생 지수)" vs "dopamine index (도파민 지수)". Tone: warm, illustrated, Spotify Wrapped-adjacent. Shareable to Instagram Stories.
2. **Algorithm Mirror mode (deeper feature).** Same data, different read: "Instagram thinks you are a 34-year-old divorced dad. Here's the receipts." Exposes the algorithm's psychological profile of the user. Friends compare not their tastes but how differently the same platforms are profiling them. Tone: mild horror, forensic readout.

The hybrid is the bet — warm onboarding lowers install friction, the Algorithm Mirror reveal creates the discourse-bait moment that drives press coverage and tech-twitter virality. The cards are the entire growth engine; nothing else matters at v0.

## Constraints

- **Side project cadence.** Single builder, evenings/weekends. Cannot afford 3-6 month death valley.
- **Success metric is contested.** Stated goal is 30k GitHub stars; hero scene is Instagram Stories virality. These two audiences are nearly disjoint in practice. Resolved by deferring the choice: prove first that any card resonates, then decide which audience to optimize for.
- **No team for visual design.** Card aesthetic is a product survival risk (Premise 2) and depends on the builder's taste alone unless help is brought in.

## Premises

1. **Friend-group cold-start is the weakest assumption.** Setlog rode on daily-life social graphs; "people who want to share content tastes" is a narrower subset. User agreed this is the most fragile premise. Mitigation in Approach C: defer friend groups entirely until card virality is proven.
2. **Card visual + insight quality is product-survival.** AI-generated cards are commodity by default. Differentiation must come from (a) visual design, (b) insight specificity (not generic horoscope-tier text), (c) tonal consistency. If all three are not landed, the product fails regardless of infra.
3. **30k GitHub stars vs Instagram-viral consumer app is a real contradiction.** Almost no precedents. Resolved (in this design) by: (a) building OSS from day one to keep the OSS lane open, (b) accepting that initial proof point is card virality on Twitter/IG, not stars, (c) reassessing the stars goal after first proof. The user defended the stars goal but accepted hybrid framing.
4. **Save-habit bootstrap is required at v1, not v0.** iOS share-sheet friction is real. At v0, the actual ingestion path is **paste 10 YouTube links manually** — that is the only path that produces a card in <60 seconds from first visit. Watch-history exports (TikTok/Instagram require a 24-hour wait for data export) are a stretch goal for v1, not v0. Premise 4 originally claimed exports sidestep friction; that was wrong — the 24-hour wait IS friction.
5. **"AI diary + dopamine/god-life index" is novel only in combination.** Adjacent: Mem.ai, Rewind, Saner.ai, Hoarder, Karakeep. The combination "content consumption + friend group + provocative card" is the genuine new angle. With Approach C, the angle simplifies to "provocative card from your watch history" — which is even more novel because no existing tool does this for short-form video.

## Approaches Considered

### Approach A: Minimum weekend build (solo experience)
Hoarder fork + iOS Shortcut + Next.js + Satori. Solo, no friend groups, two card modes. ~2 weeks. Low risk. Trades off: solo cards have a lower viral ceiling, but ships fast.

### Approach B: Full architecture (3-6 months)
Native iOS app + share extension + friend groups + OSS self-hostable backend + plugin system. Full vision. Trades off: side-project death valley risk is high; the subagent's warning ("do not build retention infra before the legend exists") applies maximally.

### Approach C: Legend-First card generator (CHOSEN)
Skip the app entirely. Build only the card generator as a web service. Users upload watch-history exports / paste screenshots / paste links. One beautiful Algorithm Mirror card is generated and shared to Twitter / Product Hunt. If the card pops, the rest of the product follows. If it doesn't, the premise is wrong and no app would have saved it. ~2 weeks (weekend prototype + weekend launch). Medium risk concentrated in a single bet.

## Recommended Approach

**Approach C: Legend-First.**

Rationale: aligns directly with the founder's stated goal ("make myself known"). Validates the riskiest premise (cards being shareable) within one week with a real signal from the open internet. Subagent's instinct ("the deliverable is one tweetable screenshot") was correct — building app infrastructure before that screenshot is proven is premature. Friend groups, share extension, native app, retention — all wait until card virality is established or denied with data.

**Concrete build sketch:**
- **Stack:** Next.js + Supabase + Claude API + Satori OR `@vercel/og` for PNG rendering. **Satori risk flagged:** Satori does not support arbitrary CSS (no `filter`, limited `background`, no `box-shadow` blur in older versions). First 2 hours of weekend 1 are a Satori prototype: if the "forensic black-and-white serif" aesthetic isn't reachable, switch to Playwright + headless Chromium on a non-Vercel runtime (Railway / Fly) before going further.
- **Ingestion v0 (single platform):** Pasted YouTube links + title (user pastes title manually if YouTube oEmbed fails). YouTube only at v0. Instagram and TikTok have no working public oEmbed for unauthenticated apps and are deferred to v1. YouTube Takeout JSON upload is a stretch goal for v0 if time permits.
- **Pipeline:** YouTube oEmbed → batch 20 items per Claude prompt for categorization (avoids Vercel 10s/60s timeout) → store tags in Supabase → card renderer outputs PNG.
- **Card mode at launch: ONE, not two.** The mockup gate (see Assignment) decides whether Warm or Algorithm Mirror ships first. The losing mode is deferred to v1. Premise 2 (visual quality is product-survival) is incompatible with shipping two modes from a single-builder team in two weekends.
- **Tag vocabularies (must define before writing prompt):**
  - **Categories:** Fashion, Self-development, Fitness, Music, Comedy/Meme, News, Gaming, Cooking, Tech, Other. (10 buckets.)
  - **Manipulation hooks (for Algorithm Mirror):** Outrage, Parasocial, FOMO, Body-comparison, Validation-seeking, Tribal-identity, Doom-scrolling, Aspirational-envy. (8 hooks.)
  - **Demographic signals (for Algorithm Mirror):** Inferred age bracket, inferred relationship status, inferred profession bucket, inferred political lean (toned down — "civic-engagement low/medium/high"). Each is a single Claude inference over the full saved set, not per-item.
- **Index formulas:**
  - **Dopamine index (0–100):** percentage of saved items tagged in `{Comedy/Meme, Outrage, FOMO, Doom-scrolling}`.
  - **God-life index (0–100):** percentage of saved items tagged in `{Self-development, Fitness, Cooking, Tech-learning, Finance}`.
  - Floor at 0 and ceiling at 100. If saved-item count <10, show "Not enough data yet — save 10+ items."
- **Card route privacy:** `/card/[userId]` is **default-private** (only authed user can view). User explicitly clicks "Publish" to mint a public slug at `/c/[shortSlug]`. The Algorithm Mirror card is literally a psychological profile; public-by-default would be a privacy disaster.
- **Auth:** Supabase magic link via Resend (NOT Supabase's default sender — KR deliverability is poor). From-domain must be configured before launch.
- **Open Source posture (deferred):** Repo is public from day one (MIT) but README narrative ("self-host your dossier"), self-host docs, and OSS badge wait until after the resonance check passes. Premature OSS narrative is scope-creep against the v0 goal.

- **Korean Z-gen reception of algorithm-criticism framing.** My position is that Algorithm Mirror beats warm-diary on virality globally, but I named the risk that Korean Z-gen culture may not respond to algorithm-paranoia the same way English-speaking tech-twitter does. The launch should include a Korean test (e.g., post both card modes to a Korean tech / lifestyle audience) before deciding which mode to lead with for influencer outreach.
- **Watch-history export availability.** TikTok and Instagram both offer data exports, but the user experience of "go to settings → request export → wait 24 hours → upload JSON to FeedDiary" is itself a major friction point. May need to lead with the lower-fidelity but immediate path: paste 10 links manually. Test ingestion friction during weekend 1.
- **Stars goal reassessment trigger.** Define a clear checkpoint: if the launch card gets >500 likes on a single Twitter post within 72 hours, evaluate building Approach B (OSS app). If it gets <100 likes, the entire premise is wrong and reposition or kill the project.

## Success Criteria

**Plan is gated. Each gate must pass before next phase. Failure at any gate triggers either pivot or kill.**

- **Gate 0 — Mockup tweet (week 0, before any code):** Two static card mockups (Warm + Algorithm Mirror) posted to Twitter with "would you use this? which mode?" caption. Threshold: **>50 likes or >5 substantive replies within 72h.** Pass = proceed to Gate 1 with the winning mode. Fail = kill or radically re-scope.
- **Gate 1 — Build (weekends 1-2):** Working v0 web app deployed, ONE card mode (the Gate 0 winner), YouTube paste-link ingestion only. Aesthetic locked. Self-test: builder generates own card and posts to own IG/Twitter.
- **Gate 2 — Resonance (week 3, after launch):** Single launch post receives **>500 likes OR >50 retweets OR Product Hunt #1-of-the-day**. Pass = continue. Fail = card premise rejected; either reposition tone, kill, or move to a different product.
- **Gate 3 — Organic share (weeks 4-8):** **A non-builder posts a card to their own Instagram Story or Twitter unprompted.** Instrument via UTM-tracked share buttons so this is countable, not anecdotal. Single most important signal in the entire plan. No-pass within 8 weeks = friend-graph extension (Approach B) probably won't save it either.
- **Gate 4 — OSS lane viability (week 8):** ≥200 GitHub stars from launch traffic alone. Pass = OSS lane is alive; promote OSS narrative and consider Approach B. Fail = drop the 30k stars goal explicitly and rewrite success as MAU / press / influencer pickup.
- **Stars milestone ladder (only relevant if Gate 4 passes):** 200 → 2k (month 3) → 10k (month 9) → 30k (month 18+). Each rung requires a corresponding action: 200=launch alone; 2k=Show HN front page + Korean press; 10k=major influencer or established OSS adoption; 30k=mainstream consumer scale OR enterprise/dev adoption. If any rung stalls for >2x its target window, reassess.

## Data Handling & Privacy

The Algorithm Mirror card is, by construction, a psychological profile. Korean PIPA and (for international users) GDPR both treat inferred profiles as sensitive personal data. Non-compliance is a real legal exposure even at v0.

- **What is stored:** Pasted URLs + extracted titles + Claude-generated tags. No video content. No watch-history beyond what the user pastes.
- **Retention:** 90 days from last user activity, then auto-deleted. User can delete on demand at any time via a `/settings/delete` endpoint.
- **LLM training opt-out:** Anthropic API by default does not train on customer data — confirm with current ToS at launch. Disclose in privacy policy.
- **Third-party data sharing:** None. No analytics SDK except Vercel's built-in (no PII), no marketing pixels.
- **Card visibility:** Default private. Public publish requires explicit user action. Public cards are accessible at unguessable slugs (UUIDs), not user IDs.
- **Korean PIPA specifics:** Privacy policy in Korean before any Korean-language launch (Gate 2+). Disclose data controller identity. Provide an email contact for data subject requests.
- **GDPR specifics:** No EU launch surface at v0 to avoid the legal-base ambiguity around "psychological profile inference." If Gate 2 passes, get a real privacy review before opening to EU traffic.
